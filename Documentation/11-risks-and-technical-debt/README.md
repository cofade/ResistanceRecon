# 11. Risks and Technical Debt

## Top risks (with mitigations)

| Risk | Mitigation |
|---|---|
| **Homology-split bug leaks clonal genomes** → inflated metrics (highest-value to get right) | Group by clonal lineage (MLST + Mash), explicit no-leakage test, report per-fold class balance |
| **BV-BRC access/filename/`evidence` vocabulary uncertainty** | Verify FTPS + filename from a real client; enumerate all distinct `evidence` values before finalizing the filter |
| **Thin per-drug label volume after grouping** | Min-n gate (≥20 R / ≥20 S) → "insufficient data" no-call rather than an unstable model |
| **AMRFinderPlus 4.2.4 `--organism` DB bug; `PARTIAL_CONTIG_END` artifacts** | Pin tag ≥ 4.2.5; flag `PARTIAL_CONTIG_END` as QC, not a real partial gene |
| **No K. pneumoniae AMRFinderPlus concordance study found** | Run our own genotype→phenotype concordance check; treat as to-validate |
| **Mock/real annotator drift** (AMRFinderPlus never in CI) | Documented periodic manual re-validation against a real WSL2/Docker run |
| **LLM-boundary shortcut under time pressure** | CI import-boundary test + CLAUDE.md golden rule |
| **Demo SPOF: Docker/WSL2/OpenAI** | Pre-computed feature cache for demo genomes; deterministic no-LLM demo path; envelope degradation |

## Known technical debt (accepted for the 24h MVP)

- Single outer grouped split instead of full nested CV (documented; variance estimates are a follow-on).
- **Calibration-population mismatch (revisit in EPIC 3 PR-B).** Per ADR-0003/decision, models train on all labelled rows but report headline metrics on the gate-negative subset they actually serve. The sigmoid calibration fold, however, is the full mixed (gate-positive + gate-negative) population, while the calibrated confidence is consumed only for gate-negative genomes at inference. Conformal (PR-B) sits on top of these calibrated probabilities. Flagged in EPIC 3 PR-A senior review; assess in PR-B whether calibration/conformal should be fit on the gate-negative fold specifically, and document the choice.
- arc42-lite (deployment-view and quality-scenario chapters dropped).
- MRSA/second species deferred.
- Thin MVP AMR-mechanism KB → many evidence items are statistical-association-only (stated in the model card).
- `ml` optional-extra needs a `numba>=0.60` floor so the resolver skips the ancient `numba 0.53.1` / `llvmlite 0.36.0` (Python <3.10 only). Revisit when `shap` tightens its own floors.
- `scripts/check_import_boundary.py` (golden rule #1's CI gate) only greps `reader/`, `features/`, `predictor/` for a direct `import genome_firewall.llm`. It does not scan `schemas.py`/`constants.py`, which EPIC 2 made the universal dependency of the trust-critical path -- a transitive LLM taint introduced via `schemas.py` would currently pass the gate undetected. Flagged in EPIC 2 senior review; not fixed there (would need an AST-based or transitive-import check, a separate, non-trivial change). Revisit before `llm/` grows real logic that could plausibly be imported from `schemas.py`.

## 11.4 Hard-won lessons from implementation

**This is the canonical, growing log of implementation lessons — the durable home the `debug-verbose` case studies and the `CLAUDE.md` "Known AI pitfalls" quick-list both point to.** Capturing a lesson here is **mandatory and non-negotiable**: every non-obvious bug and every hard-won lesson is recorded **in the same session it is learned** — no exceptions — as (1) an entry here, (2) a `debug-verbose` case study if it was instrumented, (3) a `CLAUDE.md` Known-AI-pitfall line, and (4) a pinning regression test. The senior-reviewer treats a missing capture as a P1. A lesson that lives only in chat is a lesson lost.

**House style (per entry):** a bold one-line imperative rule → the issue/PR that surfaced it → symptoms as observed → root cause with `file:function` references → the fix → the pinning test (`Pinned by tests/...`) → a generalized lesson.

Seed entries below are **standing traps** (design-time invariants, no incident yet); real incidents are appended in the same format as they occur.

- **Never give an LLM schema a verdict/confidence field.** _Standing trap._ Symptom: an LLM narrative could state a verdict the deterministic model never produced. Root cause: write access to a verdict field on an LLM output schema. Prevention/fix: LLM output schemas carry no verdict/confidence/SIR field; `scripts/check_import_boundary.py` + schema tests enforce it. Lesson: keep the LLM structurally unable to influence a verdict, not merely instructed not to.
- **Never split near-identical clonal genomes across train/test.** _Standing trap._ Symptom: inflated held-out accuracy. Root cause: a plain random split leaks near-duplicate clonal genomes across folds. Prevention/fix: homology-aware grouped split (MLST primary, Mash @ ANI 99.5% fallback) + an explicit no-leakage test. Lesson: correctness of the *split* dominates model choice for honest metrics.
- **Verify AMRFinderPlus's real TSV column/value shapes against a live run, not just the pre-implementation research doc.** EPIC 2 (#16/#17). Symptom: `research-findings/amrfinderplus-features.md`'s Method-value table listed bare names (`ALLELE`, `EXACT`, `BLAST`, `PARTIAL`, `PARTIAL_CONTIG_END`, `POINT`) and an "Element type"/"Element subtype" column-name pair; a live `docker run ncbi/amr:4.2.7-2026-05-15.1` pass against a real K. pneumoniae genome (NCBI RefSeq GCF_000016305.1) instead emitted a leading `Name` column, `Type`/`Subtype` headers (not "Element type"/"Element subtype"), and every BLAST-derived Method suffixed X/P/N (`ALLELEX`, `EXACTX`, `BLASTX`, `PARTIALX`, `PARTIAL_CONTIG_ENDX`, `POINTX`/`POINTP`/`POINTN`) per the AMRFinderPlus wiki's X/P/N-suffix convention -- the bare names never actually appear in output. Root cause: `schemas.py:AmrMethod` was written from the paraphrased research doc before any real run; had it shipped uncorrected, `annotation/_tsv.py:parse_amrfinder_tsv`'s `AmrFeature.model_validate` call would have raised `ValidationError` on every single real hit (100% failure, not a subtle drift). Fix: ran the real Docker pass during implementation rather than deferring it to the user's manual test, corrected `AmrMethod` to the full X/P/N-suffixed set, and built `tests/fixtures/amrfinder/` from the real TSV output instead of hand-guessed data. Pinned by: `tests/annotation/test_tsv.py`, `tests/reader/test_feature_builder.py` (parse the real-derived fixtures). Lesson: a pre-implementation research doc is a starting hypothesis, not ground truth for exact wire formats -- run the real tool at least once before finalizing a schema built from a paraphrase of its documentation.
- **Verify live network behavior before trusting a written protocol recipe.** EPIC 1 (#11). Symptom: `ftps_download` completed the FTPS control-channel handshake (connect/login/PROT P) and a fresh `PASV` negotiation cleanly, yet every `LIST`/`NLST`/`RETR` data transfer failed with `425 Unable to build data connection: Operation not permitted`. Root cause: `scripts/fetch_bvbrc_data.py:ftps_download` is correct per the FTPS protocol and the research doc's recipe, but this network's router (consumer NAT with an FTP ALG) cannot track FTPS's TLS-encrypted control channel and mishandles the passive-mode data connection — a network/infrastructure constraint, not a code defect. Fix: added a specific `425`/"data connection" branch to the `{ok,source,error}` hint (VPN / disable router FTP-ALG / different network) instead of leaving it under a generic message. Pinned by: `@pytest.mark.live` tests in `tests/scripts/test_fetch_bvbrc_live.py` (opt-in, `GF_RUN_LIVE=1`) exercise the real endpoint so this failure class is visible instead of only ever running against mocks. Lesson: a protocol implementation correct against RFC/docs can still fail against the *actual* network path — run the real external call at least once before trusting fixture-only tests for anything with a live-network component.
- **RQL values need form-encoding; `json(nl,map)` changes the facet response shape.** EPIC 1 (#11/#12). Symptom: a POST to `bv-brc.org/api/genome_amr/` with `eq(evidence,Laboratory Method)` (a literal space) returned `HTTP 400 Bad Request`; a separate query requesting `json(nl,map)` returned facets as `{value: count, ...}` while the client assumed Solr's default flat `[value, count, value, count, ...]` array, raising `KeyError` on slice indexing. Root cause: `scripts/fetch_bvbrc_data.py:lab_ast_facet_rql`/`evidence_vocabulary_rql` built RQL strings by naive f-string interpolation without encoding the literal space in `dataset.LAB_EVIDENCE`, and the parsing code didn't account for the `nl,map` directive it was itself requesting. Fix: added `_rql_value()` (space -> `+`) for every RQL literal; extracted a pure `_parse_facet_map()` and wired it into `cmd_crosscheck` so the evidence-vocabulary facet has a real production consumer, not just a test. Pinned by: offline unit tests in `tests/scripts/test_fetch_bvbrc_data.py` (`test_rql_value_encodes_spaces`, `test_lab_ast_facet_rql_encodes_evidence_value`, `test_parse_facet_map_matches_real_solr_shape` — the last against a payload matching the exact shape returned by the live API) that run in every `pytest` invocation, not only the CI-skipped live tests that first caught it. The live tests (`test_solr_evidence_facet_has_lab_method`, `test_solr_numfound_nonzero`) confirmed `numFound=85291` against the real endpoint, matching the research doc's independently-recorded figure exactly, and remain a supplementary real-network check. Lesson: when building any query DSL by string interpolation, either use a proper encoder or write a live test exercising a value with the exact special character (space, `/`, etc.) expected in production data — `"Laboratory Method"` is the single most important literal string in the whole pipeline (ADR-0001) and its encoding was untested until run live.
- **Always bound and sanity-check a paged HTTPS API download; a default row-limit truncates silently.** EPIC 3 (#18, ADR-0016). Symptom: the BV-BRC Data API's `genome_sequence` collection returns only 25 contigs by default (page cap 25,000), so a naive query for a multi-contig K. pneumoniae assembly would write a FASTA missing most of the genome with a `200 OK` and no error — every downstream AMRFinderPlus feature would then be computed on a fragment. Root cause: REST record APIs paginate; the FASTA representation carries no total-count signal, so a partial response is indistinguishable from a complete one without an external check. Fix: `scripts/fetch_bvbrc_data.py:https_fasta_download` requests an explicit `limit(25000)` and `fasta_sanity_problem` validates the returned contig count / total length against the `genome` record's `contigs`/`genome_length`, rejecting on mismatch or on reaching the limit ceiling. Pinned by `tests/scripts/test_fetch_https_helpers.py` (offline) and `tests/scripts/test_fetch_https_live.py` (opt-in, against the real API). Lesson: for any external paged/streamed download, a `success` status is not evidence of completeness — verify size against an independent record before trusting the payload.
- **Verify a third-party library's current API against the installed version, not the ADR's recipe.** EPIC 3 (#20, ADR-0003/0004). Symptom: implementing the LR + sigmoid-calibration recipe verbatim produced a `FutureWarning` (`penalty='l2'` deprecated in sklearn 1.8) and, on sklearn 1.9, `CalibratedClassifierCV(cv='prefit')` is gone entirely. Root cause: scikit-learn evolved its API after ADR-0003/0004 were written — L2 is now the unnamed default and prefit calibration is expressed by wrapping the base model in `sklearn.frozen.FrozenEstimator`. Fix: `predictor/train.py` relies on `LogisticRegression`'s default L2 (dropping the explicit `penalty='l2'`); `predictor/calibration.py` uses `CalibratedClassifierCV(FrozenEstimator(model), method='sigmoid')` with a fallback to `cv='prefit'` for older sklearn. A **second, imbalanced-only** warning surfaced in senior review: with a frozen base the calibrator still runs an internal `StratifiedKFold(5)`, which emits `"least populated class has only N members < n_splits"` whenever a per-drug calibration fold's minority class has <5 members (routine for carbapenemase-heavy meropenem). Because predictions are fold-independent under `FrozenEstimator`, the fix is to bound that internal `cv` to the minority-class size (`max(2, min(5, minority))`) -- same calibrator, no warning. Pinned by `tests/predictor/test_calibration.py::test_calibrate_is_warning_free_on_an_imbalanced_fold` and the meropenem arm of the training e2e. Lesson: an ADR fixes the *method*; the *exact call* must be re-checked against the resolved dependency version AND on the imbalanced/clonal data shape (not just the balanced happy path), and a deprecation/warning is a signal to adapt deliberately (and record it) rather than suppress or leave to the happy path.
- **Gate the grouped split on distinct-group count, not only R/S count; a clonal drug crashes StratifiedGroupKFold.** EPIC 3 (#18, senior review). Symptom: a drug clearing the min-n gate (≥20 R / ≥20 S) but whose labels concentrate in <4 MLST STs raised an uncaught `ValueError: Cannot have number of splits n_splits=2 greater than the number of groups: 1` from `StratifiedGroupKFold` (via `predictor/split.make_split` → `leave_one_group_out_holdout` leaving one group → `three_way_grouped_split` on a single remaining group). Exactly the real-data shape (ST258/ST512/ST11 dominance). Root cause: `safe_n_splits` floors at 2 (StratifiedGroupKFold's minimum) even when the group count is 1, and `make_split` didn't check group diversity before splitting. Fix: `make_split` now gates `n_groups >= MIN_DISTINCT_GROUPS (=4)` and wraps the split construction in `try/except ValueError`, returning `insufficient_data` (a genuine `LeakageError` is a `RuntimeError` and is NOT swallowed). Pinned by `tests/predictor/test_split.py::test_make_split_reports_insufficient_data_on_a_too_clonal_drug`. Lesson: a min-n gate on class counts is necessary but not sufficient for a *grouped* split — the group count is a second, independent feasibility constraint, and the failure mode is a crash on precisely the clonal data the split exists to handle.
- **A sentence-transformers embedder downloads model weights on first construction — never build it in CI.** EPIC 5 (#24, ADR-0019). Symptom: any test that constructs `SentenceTransformer(...)` (directly or via a module-level dense retriever) would fetch weights from the network on first use, which the autouse `_no_network` guard turns into a hard failure — guaranteed red CI on an otherwise-correct RAG. Root cause: the heavy semantic model is a network-dependent external resource, the same class of problem as AMRFinderPlus (ADR-0002), but easy to trigger implicitly because embedding "feels" like pure computation. Fix: the dense leg sits behind an `Embedder` Protocol; CI/tests inject `HashingBagOfWordsEmbedder` (deterministic, numpy-only, BLAKE2b token hashing — not Python's per-process `hash`, which would break cross-run determinism); `SentenceTransformerEmbedder` constructs the model lazily inside `embed` and is never built in CI. Pinned by `tests/kb/test_no_sentence_transformer_construction.py` (monkeypatches `sentence_transformers.SentenceTransformer` to raise, then runs the full RAG path) and `tests/kb/test_embedder.py` (constructing the wrapper leaves `_model is None`). Lesson: treat any model-weight-loading class exactly like a subprocess/Docker dependency — isolate it behind an interface with a deterministic offline stand-in, and assert *by construction* that CI never instantiates the real one.
- **Anchor the row-level evidence_category to strongest-cited-item, not to gate firing — the gate-anchored rule trips the schema validator on a lone KB-member gene.** EPIC 5 (#23, ADR-0020). Symptom: an early tagging rule set a row's `evidence_category` to `known_mechanism` **iff the deterministic gate fired**, else `statistical_association`. For a genome carrying a single curated known-mechanism gene that does *not* fire the gate (e.g. a lone `qnrB`, or an AME on gentamicin), the builder would cite a `known_mechanism` EvidenceItem while labelling the row `statistical_association` — which raises `AntibioticPrediction`'s validator #3 ("row category must be backed by a cited item of that category"), crashing report construction on real data. Root cause: gate firing (which *forces a verdict*) and KB membership (which *characterises evidence*) are different axes; conflating them makes the row category disagree with its own cited items. Fix: per-item category is KB-membership-based; the row category is the **strongest** category among cited items (`known_mechanism` > `statistical_association` > `no_signal`), which satisfies all three validators by construction while keeping confidence independent (gate hit -> 0.99; otherwise the calibrated model value, even for a `known_mechanism` row). Pinned by `tests/report/test_builder_validators.py` (every genome×drug in the synthetic cohort constructs without `ValidationError`) and `tests/report/test_evidence.py`. Lesson: when a Pydantic cross-field validator ties a summary field to its detail items, derive the summary *from* the items (a max/rollup), never from a parallel signal that can disagree with them.
- **Close the invariant CLASS, not one field — a soft "this prose is about one drug" assumption, hardened in one field, re-opens the same hole in the next.** EPIC 5 (#26, senior review rounds 1-4). Symptom: the LLM-narrative reviewer's deterministic pre-check leaked a *wrong verdict for a drug into the clinician-facing `narrative_summary`* four times, each in a different prose surface: (1) a global verdict-word membership test passed a per-drug swap in a mixed-verdict panel; fixed to per-`per_antibiotic` binding — but (2) the *summary/caveats* still used the flattened test; unified to a per-field proximity check — but (3) *proximity attribution* bound each phrase to the single nearest drug, so a plural sentence ("meropenem and ceftriaxone are both likely to work") masked the farther drug's wrong verdict; the free-text summary/caveats were then *banned* from stating any verdict/causal phrase — but (4) the identical plural/aggregate attack simply moved *inside a `per_antibiotic` narrative*, which was checked only against its own drug's verdict while free to name a second evaluated drug. Root cause: every round hardened one *instance* while leaving the unstated invariant — "a piece of prose attributes its verdict/causal claims to exactly the drug(s) it is contractually about" — soft and load-bearing in the next surface. Proximity/membership are heuristics masquerading as a deterministic guarantee. Fix: make the *contract* unambiguous instead of the parser smarter — a `per_antibiotic` narrative may name no other panel drug (attribution is then provably its own), and the free-text summary/caveats may state no verdict/causal phrase at all (they belong in the per-drug narratives). Both banned outright, symmetric, no phrase-to-drug guessing anywhere. Pinned by `tests/report/test_reviewer.py` (per-drug swap, summary swap, caveat swap, plural-in-summary, foreign-drug-in-entry, foreign-causal-in-entry). Note the structural spine held throughout: golden rule #1 (no verdict/confidence *field* on any LLM schema) was never breached — what leaked was wrong prose, caught only because the fail-closed pre-check is treated as the reliable guard (the LLM judge shares the narrator's failure mode). Lesson: when a check depends on an assumption about the *shape* of its input, enforce that shape as a structural invariant across every symmetric field at once and add a regression test per field — hardening field A while leaving the same assumption in field B ships the same bug again, and "the commit says it's closed" is not "the code closed it." A relocatable attack is not fixed until the class is closed.
